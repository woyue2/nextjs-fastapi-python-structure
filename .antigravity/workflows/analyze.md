## 用途

每次接到新需求时，**第一步**执行此工作流。
通过评分机制验证需求完整性，避免"需求理解偏差"导致的无效开发。

---

> 让 AI 在开始分析需求前，先知道完整的开发轨道规则。
> 这一步将设计阶段、实现阶段等所有节点约束加入上下文，
> 防止 AI 分析完直接跳入实现，跨越 `/design` 拍板节点。

## 执行流程总览

```
阶段A (步骤 1-4) → 关键检查点: 评分 ≥ 7分?
  ├─ 是 → 执行阶段B (步骤 5-6) → 输出需求分析总结 → 等待下一步
  └─ 否 → 输出追问 → 等待用户补充 → 重新评分
              └─ 循环，直到 ≥7分 或 用户取消
```

⚠️ **关键约束：评分 < 7 分时，禁止执行阶段B，禁止输出总结，只输出追问。**

## 阶段A：需求评估

### 步骤 1 — 检查知识库 / 代码库状态

```yaml
判定条件: 工作目录中存在代码文件 且 需求不是"新建项目"
执行方式:
  - 优先检查是否存在知识库（如 CLAUDE.md、docs/、.antigravity）
  - 若知识库不完整 → 扫描代码库结构（list_dir / grep_search）
  - 知识库缺失时：仅标记问题，不创建（只读，P1 阶段不写文件）
```

// turbo
1. 列出项目根目录结构
   CommandLine: dir /b

// turbo
2. 列出 src/ 目录结构（若存在）
   CommandLine: if exist src dir /b /s src

### 步骤 2 — 获取项目上下文

```yaml
目的: 为评分和追问提供完整项目上下文，避免询问可从代码推断的信息
执行方式:

  - 不足则扫描代码库（查看 package.json、requirements.txt、主入口文件）
  - 记录技术栈、目录结构、已有模块
```

### 步骤 3 — 需求类型判定

判定需求属于以下哪种类型（影响后续分析深度）：

| 类型 | 说明 | 示例 |
|------|------|------|
| 新项目初始化 | 从零搭建 | "帮我初始化一个 Next.js 项目" |
| 重大功能重构 | 影响多模块 | "把邀请码改为手机号绑定" |
| 常规功能开发 | 新增独立功能 | "新增用户头像上传" |
| Bug 修复 | 定位并修复 | "登录后跳转异常" |
| 技术变更 | 框架/依赖/配置 | "升级 TailwindCSS v4" |
| 文档 / 重构 | 不改业务逻辑 | "重构 Button 组件" |

---

### 步骤 4 — 需求完整性评分【关键检查点】

在 `<thinking>` 中（不输出给用户）完成以下推理：

```
<thinking>
逐项评分（总分 10 分）：
  - 目标明确性 (0-3分): 任务目标是否清晰具体 → X分
  - 预期结果   (0-3分): 成功标准和交付物是否明确 → X分
  - 边界范围   (0-2分): 任务范围和边界是否清楚 → X分
  - 约束条件   (0-2分): 时间/性能/业务限制是否说明 → X分
总分: X/10
缺失信息: [列举]
判定: 追问 or 继续
</thinking>
```

**评分追问规则：**
- ✅ 严格避免询问已知信息（技术栈、框架、可从代码推断的内容）
- ❓ 只询问用户相关信息：具体需求、业务逻辑、期望结果、优先级、约束条件

---

### 追问输出格式（评分 < 7 分时）

行首必须为：`❓【需求分析】`

```
❓【需求分析】

当前需求完整性评分为 X/10 分，[一句话说明缺失什么]。

1. [问题1]
2. [问题2]
3. [问题3]（最多5个，扁平化，不分类）

请按序号回答，或输入"以现有需求继续"跳过追问（可能影响方案质量）。
```

**用户响应处理：**

| 用户输入 | 处理方式 |
|---------|---------|
| 补充信息 | 重新评分 → 继续循环 |
| "以现有需求继续" | 直接执行阶段B，不再追问 |
| "取消" | 输出取消格式，清理状态 |

取消格式：
```
🚫【需求分析】已取消

已取消: 需求分析
────
🔄 下一步: 可重新描述需求或进行其他操作
```

## 阶段B：代码分析（仅评分 ≥ 7 分后执行）

### 步骤 5 — 提取关键目标与成功标准

- **关键目标**：从需求中提炼 1-3 个核心目标（用动词开头）
- **成功标准**：定义可验证的完成条件（可量化 / 可演示）

示例：
```
关键目标:
  1. 新增 /api/users/bind 接口，支持手机号匹配绑定
  2. 前端绑定表单从邀请码输入改为手机号输入

成功标准:
  - 后端接口返回 200，绑定关系写入数据库
  - 前端表单校验手机号格式（11位，1开头）
  - 原有邀请码逻辑可软删除（不影响历史数据）
```

---

### 步骤 6 — 代码分析与技术准备

```yaml
执行内容:
  定位相关模块:
    - 前端: 找到对应页面/组件文件（frontend/src/app/...）
    - 后端: 找到对应路由文件（backend/routers/...）
  质量检查:
    - 标记过时依赖或 API
    - 扫描安全风险（如明文密码、未校验输入）
    - 识别代码异味（重复代码、超长函数、深层嵌套）
  技术信息收集:
    - 如涉及第三方库，确认当前版本与最新 API
    - 如涉及数据库，确认 Schema 结构
```
