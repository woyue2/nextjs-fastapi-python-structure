## 用途

遇到 Bug、报错、验收失败时，**先跑此工作流**。
目标：穷举失败原因并排序，为 `/design fix` 提供输入。

⚠️ **严格约束：禁止给出任何修复方案、禁止写代码，只做分析。**

---

## 输入准备（运行前请提供）

```
📋 请提供以下信息（有多少提多少）：
  1. 出错的代码片段（或文件路径）
  2. 完整报错信息 / 错误日志
  3. 验收失败的具体现象（用户看到了什么）
  4. 复现步骤（如果知道）
  5. 最近一次正常工作时的状态（如：哪次 commit 之前是好的）
```

## 阶段B：根因分析

### 步骤 1 — 失败原因穷举

穷举 **所有可能的失败原因**，思维顺序：

```
1. 直接原因  ← 报错信息明确指向的位置
2. 间接原因  ← 调用链上游的问题
3. 环境原因  ← 依赖版本、环境变量、运行时差异
4. 时序原因  ← 异步顺序、竞态条件、缓存状态
5. 边界条件  ← 空值、空数组、超长字符串、并发请求
6. 隐含假设  ← 代码中"应该总是这样"但未做保护的地方
```

### 步骤 2 — 概率标注

对每条原因标注可能性：

| 标记 | 含义 |
|------|------|
| 🔴 高概率 | 直接证据支持，大概率是根因 |
| 🟡 中概率 | 间接迹象，需要验证 |
| 🟢 低概率 | 理论可能但现象不匹配 |
| ❓ 未知   | 没有足够信息判断 |

### 步骤 3 — 信息缺口声明

明确列出：

```
目前信息不足的地方：
  - [缺少 X 信息，无法判断 Y 原因]
  - [需要查看 Z 文件才能确认 W 原因]
```

## 输出格式（固定，不得随意更改）

```
🔍【问题诊断】

📌 问题描述: [一句话复述用户描述的现象]
📁 涉及位置: [文件路径 / 模块名]

───────────────────────────────
💥 可能的失败原因：

🔴 高概率
  1. [原因] — [根据的证据]
  2. [原因] — [根据的证据]

🟡 中概率
  3. [原因] — [间接迹象]

🟢 低概率
  4. [原因] — [理论可能]

❓ 信息不足，无法判断
  5. [原因] — [需要什么信息才能判断]

───────────────────────────────
📭 当前信息缺口：
  - [缺少 X，无法确认 Y]
  - [需要 Z 才能排除 W]

───────────────────────────────
🚫 本阶段不提供任何修复方案。
下一步: 运行 /design fix 进入修复方案设计
```

## 与其他工作流的关系

```
debug.md（根因分析）
    └─ 分析完成后
        └─ /design fix → design.md（修复模式）
                └─ 方案确认后 → /implement（内联 review+sync_doc）
                        └─ /git_commit
```
