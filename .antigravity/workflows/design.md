## 用途

接在 `analyze.md`（功能开发）或 `debug.md`（Bug 修复）之后执行。
目标：提出 ≥3 个备选方案，对比权衡，给出推荐，**不写具体代码**。

⚠️ **严格约束：禁止写任何代码实现。只输出方案描述、影响分析和推荐理由。**

---

## 模式判定（自动识别，无需手动指定）

| 入口 | 模式 | 上下文来源 |
|------|------|-----------|
| 接在 `/analyze` 之后 | **功能模式** | 读取 analyze 的需求总结 |
| 接在 `/debug` 之后   | **修复模式** | 读取 debug 的根因分析 |
| 独立调用             | **通用模式** | 请求用户提供背景信息 |

## 阶段A：获取上下文

### 功能模式

从上一步 `/analyze` 的输出中提取：
- 需求类型、关键目标、成功标准
- 涉及的模块路径
- 已知技术约束

### 修复模式

从上一步 `/debug` 的输出中提取：
- 🔴 高概率根因（作为方案设计的基准）
- 涉及的文件路径
- 信息缺口（方案中需要规避的不确定性）

---

## 阶段B：方案设计（核心）

### 要求

- **至少 3 种方案**，且方案之间必须有实质性差异（不是同一方案的微调）
- 差异维度举例：改动层级不同、技术路线不同、取舍侧重不同、影响范围不同
- 每个方案沿以下 3 个维度分析：

```yaml
改动范围:
  - 涉及哪些文件 / 模块 / 层级
  - 是前端 / 后端 / 全栈
  - 是局部修改还是结构性变更

风险点:
  - 引入新问题的可能性
  - 对性能/安全/可维护性的潜在影响
  - 技术债务

对现有逻辑的影响:
  - 哪些已有功能可能受影响
  - 是否需要迁移数据 / 通知前端 / 更新文档
  - 向后兼容性
```

---

## 阶段C：推荐与理由

从 ≥3 个方案中推荐最稳妥的一个，说明：

```yaml
推荐方案: 方案 X
推荐理由:
  - 为什么比其他方案更稳妥
  - 哪个风险维度得分最优
  - 接受了什么代价（没有完美方案）
前提条件:
  - 推荐成立的假设条件
  - 如果 X 条件不成立，则改选方案 Y
```

## 方案质量检查清单（输出前自检）

```
□ 方案数量 ≥ 3
□ 方案之间有实质性差异（不是同一方案的微调）
□ 每个方案都分析了：改动范围 / 风险点 / 对现有逻辑的影响
□ 推荐方案有具体理由（不是"综合考虑，方案X最好"）
□ 没有写任何代码
□ 推荐的成立前提已声明
```
