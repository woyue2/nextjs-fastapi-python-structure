## 用途

接在 `/implement` 完成之后、`/sync_doc` 同步文档之前。
目标：用 Linus 的品味标准审视刚写的代码，拦截坏味道，确保 GEB 合规。

⚠️ **不是事后复盘，是提交前的最后关卡。发现问题就在这里修。**

---

## 阶段A：品味自检（Good Taste Check）

对本次变更的每个文件，逐条检查：

### 检查清单

```yaml
分支检查:
  □ 有没有可以消除的 if/else？
  □ 是否存在 >3 个分支的逻辑？ → 立即重构或提出建议
  □ 边界条件是否自然融入常规路径（而非特殊处理）？

函数检查:
  □ 每个函数是否只做一件事？
  □ 有没有超过 20 行的函数？ → 标记并拆分
  □ 缩进是否超过 3 层？ → 标记并提取

文件检查:
  □ 单文件是否超过 800 行？ → 必须拆分
  □ 同一目录是否超过 8 个文件？ → 考虑子目录拆分

命名检查:
  □ 变量/函数名是否简洁直白？
  □ 有没有只看名字就能猜错用途的命名？
```

## 阶段B：代码坏味道扫描

对本次变更涉及的文件，检测以下 7 种坏味道：

| 坏味道 | 检测方式 | 发现后处理 |
|--------|---------|-----------|
| **僵化** | 改一处需要连锁改多处？ | 提出解耦方案 |
| **冗余** | 相同逻辑在多处重复？ | 提取公共函数/组件 |
| **循环依赖** | A 依赖 B，B 依赖 A？ | 引入中间层或接口 |
| **脆弱性** | 改此文件可能意外破坏其他文件？ | 标记影响范围 |
| **晦涩性** | 读代码需要猜测意图？ | 加注释或重命名 |
| **数据泥团** | 3+ 个参数总一起出现？ | 组合为对象/类型 |
| **过度设计** | 为不存在的需求写了抽象层？ | 删除，回归简单 |

### 发现坏味道后的输出格式

```
🦨【代码坏味道】

  位置: [文件:行号]
  类型: [7 种之一]
  现象: [具体描述]
  建议: [修复方向]
  是否立即修复？（是/否/记录到 TODO）
```

## 阶段C：GEB 合规检查

// turbo
**先获取客观依据（不可幻觉）**
CommandLine: git diff --name-only HEAD

// turbo
**查看新建文件列表**
CommandLine: git diff --diff-filter=A --name-only HEAD

// turbo
**查看被删除文件列表**
CommandLine: git diff --diff-filter=D --name-only HEAD

> 上面命令的输出是后续所有 GEB 检查的基准。对照这些文件进行逐一检查，而不是靠记忆。

### C1 — L3 契约完整性

对上面命令输出的每个文件，逐一打开检查：

```yaml
检查项:
  □ 文件头部是否有 L3 契约注释？
    - 缺失 → FATAL-002，立即补充，阻塞级优先
  □ [INPUT] 与实际 import 是否匹配？
  □ [OUTPUT] 与实际 export 是否匹配？
  □ [POS] 描述是否准确（所属模块 + 角色 + 与兄弟文件关系）？
    - 缺失 → 立即补充: [PROTOCOL]: 变更时更新此头部，然后检查 CLAUDE.md
```

### C2 — L2 成员一致性

```yaml
检查所有涉及文件增删的目录:
  □ 该目录的 CLAUDE.md 是否存在？
    - 不存在 → FATAL-004，立即创建
  □ 成员清单是否包含本次新增的文件？
    - 缺失 → FATAL-003（反向），立即添加
  □ 是否有已删除但清单仍残留的条目？
    - 残留 → FATAL-003，立即清除
```

### C3 — L1 全局一致性

```yaml
仅在以下情况触发:
  - 新增/删除了顶级模块目录
  - 技术栈发生变化（新增依赖、更换框架）
检查:
  □ 根 CLAUDE.md 的 <directory> 列表是否更新？
  □ 技术栈描述是否准确？
```

## GEB 违规输出格式

```
⛔【GEB 违规】

  代码: [FATAL-00X / SEVERE-00X]
  位置: [文件/目录路径]
  描述: [违规内容]
  修复: [立即执行的修复步骤]
  状态: ✅ 已修复 / ❌ 待修复
```

---

## 审查完成输出格式（固定）

```
✅【代码审查完成】

🎨 品味自检:
  [通过 / X 处需要调整（列出）]

🦨 坏味道扫描:
  [未发现 / 发现 X 处（已修复 Y 处，记录 Z 处 TODO）]

📜 GEB 合规:
  L3 契约: [全部完整 / X 处已补充]
  L2 成员: [全部一致 / X 处已更新]
  L1 全局: [无需更新 / 已更新]

📁 审查期间的额外文件变更:
  ~ [文件路径]  ← [变更原因：补充 L3 / 拆分函数 / ...]

下一步: /sync_doc → /git_commit
```
