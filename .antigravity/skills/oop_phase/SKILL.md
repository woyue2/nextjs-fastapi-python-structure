# OOP Phase Skill — 三期设计约束

读到此文件后，在每次涉及**结构设计、新增模块、类设计、重构**时，
自动根据项目阶段套用对应约束。**不需要用户提醒，主动判断并告知当前适用约束集。**

---

## 如何判断当前项目阶段

在开始任何设计前，先看以下信号：

```
探索期信号：
  - git log 少于 10 次提交
  - core 类少于 5 个
  - 没有稳定的数据库 Schema
  - 需求还在变化中

增长期信号：
  - 功能列表基本确定
  - 核心流程跑通，在加 feature
  - 有稳定的 API 接口但还在迭代

冻结期信号：
  - 需求冻结，只做 bugfix 和优化
  - 核心业务流程 3 个月以上无大改
  - 有完整测试覆盖
```

---

## 探索期约束（能跑 + 易改）

**目标：降低重构成本**

```
约束（必须遵守）：
  1. 不预设未来需求
  2. 不为"可能扩展"而设计
  3. 不使用设计模式
  4. 不使用抽象基类
  5. 优先组合，不使用继承
  6. 核心类不超过 5 个

结构要求：
  - 修改时影响面尽量小
  - 允许数据库字段变更
  - 不让测试依赖 ORM 结构

输出：
  - 推荐目录结构
  - 每层职责说明
  - 哪些现在不要做
  - 未来升级路径（只说明，不实现）

检查：如果新增结构，必须说明它现在必须存在的理由。
      如果超过 5 个核心类，主动简化。
```

---

## 增长期约束（保持弹性 + 控制膨胀）

**目标：功能增加但结构不腐烂**

```
约束（必须遵守）：
  - 优先修改现有结构
  - 控制核心类数量（不超过上期 +2）
  - 避免抽象基类
  - 避免继承层级
  - 不为未来假设需求设计
  - 让每个类可以独立修改

新增结构时：
  - 必须解释它解决的具体问题
  - 必须说明为什么不能通过扩展现有类实现

输出格式：
  （前面分析了很多...）分点列出要实现的东西以及步骤。
  每步说明：改哪里 → 为什么这样改 → 影响范围。
```

---

## 冻结期约束（可维护性 + 结构清晰度）

**目标：降低理解成本**

```
约束（必须遵守）：
  1. 依赖方向必须单向
  2. 业务规则必须集中（不散落在 utils/helper）
  3. IO 与领域逻辑必须分离
  4. 可以使用轻量抽象，但必须说明存在理由
  5. 不允许为"假设需求"设计

输出：
  - 推荐分层结构
  - 每层职责
  - 哪些地方可以封装为类
  - 哪些地方保持简单函数更好
  - 结构升级的边界

检查：如果新增抽象，必须解释它解决的具体问题。
```

---

## 核心认知（内化为本能）

```
真正让系统更健壮的，是"约束"，不是"类"。

探索期：降低重构成本
冻结期：降低理解成本

类不是目标，约束才是。
```

---

## 激活行为

每次被要求做设计决策时：
1. **先判断阶段**（说出依据）
2. **报告适用约束集**
3. **在约束内给出方案**
4. **如果用户要求违反约束，明确告警后才执行**
