# Anti-Huihuan 前后端目录规范

> **用途**：每次 `/implement` 或 `/review` 时，对照本文档检查目录结构是否合规。
> **最后更新**：2026-03-03

---

## 一、Backend（FastAPI）目录规范例子

### 1.1 目标结构例子，仅供参考

```
backend/
├── app/
│   ├── __init__.py
│   ├── main.py                    # 入口，只做 app = FastAPI() + include_router + middleware
│   ├── core/                      # 全局配置
│   │   ├── __init__.py
│   │   ├── config.py              # 环境变量、CORS origins 等配置
│   │   ├── security.py            # 认证/鉴权（如需要）
│   │   └── logging.py             # 日志配置
│   ├── api/
│   │   └── v1/
│   │       ├── __init__.py
│   │       ├── router.py          # 汇总所有 endpoints 的 APIRouter
│   │       └── endpoints/
│   │           ├── __init__.py
│   │           ├── health.py      # GET /api/v1/health
│   │           ├── records.py     # GET/POST /api/v1/records
│   │           └── hello.py       # GET /api/v1/hello（演示）
│   ├── schemas/                   # Pydantic 请求/响应模型
│   │   ├── __init__.py
│   │   └── record.py             # PrizeSchema, RecordCreateSchema, RecordSchema, RecordListResponse
│   ├── services/                  # 业务逻辑层（唯一写业务的地方）
│   │   ├── __init__.py
│   │   └── record_service.py     # 记录的 CRUD 逻辑、内存存储、锁
│   ├── models/                    # ORM 模型（未来接入数据库时使用）
│   │   └── __init__.py
│   ├── db/                        # 数据库 session（未来使用）
│   │   └── __init__.py
│   └── dependencies/              # FastAPI 依赖注入
│       └── __init__.py
├── requirements.txt
├── CLAUDE.md                      # GEB L2 文档
└── .venv/                         # 虚拟环境（不提交 git）
```

### 1.2 铁律

| 编号 | 规则 | 说明 |
|------|------|------|
| B-01 | **`main.py` 只做挂载** | 只允许 `app = FastAPI()`、`include_router`、`add_middleware`，禁止写业务逻辑、Schema 定义、数据操作 |
| B-02 | **路由层禁止写业务** | `endpoints/` 中的函数只做：参数验证 → 调用 service → 返回响应 |
| B-03 | **路由层禁止直接操作数据** | 不允许在 `endpoints/` 里直接操作数据库/内存存储 |
| B-04 | **API 版本管理** | 使用 `APIRouter` + `/api/v1/` 前缀分组 |
| B-05 | **Schema 与 Model 分离** | Pydantic 模型放 `schemas/`，ORM 模型放 `models/`，不混用 |
| B-06 | **无嵌套 backend/** | `backend/` 下直接是 `app/`，禁止出现 `backend/backend/` 嵌套 |

### 1.3 日志规范

```python
# ✅ 正确做法
import logging
logger = logging.getLogger(__name__)

logger.info("开始处理", extra={"url": url})
logger.error("处理失败", exc_info=True)

# ❌ 禁止
print(...)                    # 禁止 print
except Exception: pass        # 禁止吞异常
except Exception: return {}   # 必须先 logger.error 再 return
```

**必须覆盖的日志上下文**：
- 入口参数（URL、请求体摘要）
- 关键中间结果（解析条数、处理耗时）
- 最终输出（返回条数）
- 所有 `except` 分支（必须 `exc_info=True`）

### 1.4 检查清单

```
□ backend/app/main.py 是否只做挂载？（≤ 30 行业务无关代码）
□ endpoints/ 里是否有业务逻辑？
□ schemas/ 是否包含所有 Pydantic 模型？
□ services/ 是否存在且包含业务逻辑？
□ 路由前缀是否使用 /api/v1/？
□ 是否有 logger 而不是 print？
□ 所有 except 是否有 exc_info=True？
□ 是否存在 backend/backend/ 嵌套？（应删除）
```

## 二、Frontend（Next.js App Router）目录规范例子

### 2.1 目标结构例子，仅供参考

```
frontend/src/
├── app/
│   ├── layout.tsx             # Server Component · 导出项目 metadata
│   ├── page.tsx               # 首页（Server Component 优先）
│   ├── globals.css            # 全局样式 + 主题变量
│   ├── loading.tsx            # 全局 loading fallback
│   ├── error.tsx              # 全局错误边界
│   ├── lottery/
│   │   ├── page.tsx           # 抽奖页（≤ 150 行）
│   │   ├── loading.tsx        # 路由级 loading
│   │   ├── error.tsx          # 路由级错误边界
│   │   └── components/
│   │       ├── LotteryGrid.tsx    # 九宫格（≤ 150 行）
│   │       ├── ActionBtn.tsx      # 底部动作按钮
│   │       ├── WinModal.tsx       # 中奖弹窗
│   │       ├── RedeemModal.tsx    # 兑换码弹窗
│   │       ├── RecordModal.tsx    # 记录弹窗
│   │       ├── ThemeSwitcher.tsx  # 主题切换（≤ 150 行）
│   │       └── ThemePanel.tsx     # 主题面板 UI（从 ThemeSwitcher 拆出）
│   └── admin/
│       ├── page.tsx           # 后台页（≤ 150 行）
│       ├── loading.tsx        # 路由级 loading
│       ├── error.tsx          # 路由级错误边界
│       └── components/
│           ├── PrizeEditor.tsx    # 奖品编辑器（≤ 150 行）
│           ├── PrizeRow.tsx       # 单行奖品编辑（从 PrizeEditor 拆出）
│           └── ChanceEditor.tsx   # 次数编辑器
├── lib/                       # 公共工具层
│   ├── types.ts               # Prize, WinRecord, PrizeType 等共享类型
│   ├── constants.ts           # PRIZES, REDEEM_CODES 等常量数据
│   ├── api.ts                 # fetchRecords, postRecord 等 API 调用
│   └── storage.ts             # safeGet, safeSet 等 localStorage 封装
└── hooks/                     # 公共 Hooks
    ├── useLotteryStore.ts     # 抽奖状态管理（纯 Hook，不含类型/常量/API）
    └── useLotteryAnimation.ts # 九宫格动画逻辑（从 LotteryGrid 提取）
```

### 2.2 铁律

| 编号 | 规则 | 说明 |
|------|------|------|
| F-01 | **组件 ≤ 150 行** | 单个 `.tsx` 文件不超过 150 行（含 JSX），超出则拆分子组件 |
| F-02 | **`'use client'` 最小化** | 默认 Server Component，只有使用 `useState`/`useEffect`/事件监听/浏览器 API 时才加 |
| F-03 | **`layout.tsx` 禁止 `'use client'`** | 根 layout 必须是 Server Component |
| F-04 | **每个路由必须有 `loading.tsx` + `error.tsx`** | 防止 Suspense 缺失和页面白屏 |
| F-05 | **每个 `page.tsx` 必须导出 `metadata`** | SEO 基本要求，不允许用脚手架默认值 |
| F-06 | **Props 必须有接口定义** | 禁止 `props: any`，超过 3 个字段的不允许内联匿名类型 |
| F-07 | **禁止 `any`（含 `as any`）** | 用 `unknown` + 类型收窄替代 |
| F-08 | **`useState` 泛型显式标注** | `useState<Prize \| null>(null)` ✅ · `useState(null)` ❌ |
| F-09 | **内部跳转用 `next/link` 或 `useRouter`** | 禁止 `window.location.href`（除非必须强制刷新） |
| F-10 | **禁止生产代码 `console.log`** | 调试完必须删，错误用 `console.error` |
| F-11 | **共享类型/常量/工具提升到 `lib/`** | 禁止跨路由通过 `../../lottery/store` 导入，防止路由耦合 |
| F-12 | **Hook 必须以 `use` 开头** | 事件处理函数以 `handle` 开头 |

### 2.3 `'use client'` 边界决策表

| 文件类型 | 默认 | 加 `'use client'` 条件 |
|----------|------|------------------------|
| `layout.tsx` | Server | **永远不加** |
| `page.tsx` | Server | 有 `useState` / `useEffect` / 事件监听时加 |
| `components/*.tsx` | Server | 有交互逻辑时加，尽量推迟到叶子节点 |
| `store.ts` / `hooks/*.ts` | — | **必须加** |

### 2.4 内联样式约束

```
≤ 5 个属性 → 可以直接 style={{ ... }}
> 5 个属性 → 提取为具名常量：
    const cardStyle: React.CSSProperties = { ... };
    <div style={cardStyle}>
```

### 2.5 React Hooks 规范

```typescript
// ✅ useEffect 异步写法
useEffect(() => {
    const load = async () => { ... };
    load();
}, [deps]);

// ❌ 禁止
useEffect(async () => { ... }, []);     // 会吞错误
useEffect(() => { ... }, []);           // 空依赖需确认真的只运行一次

// ✅ 传给子组件的函数用 useCallback
const handleSpin = useCallback(() => { ... }, [deps]);
```

### 2.6 检查清单

```
□ 每个 .tsx 文件是否 ≤ 150 行？
□ 每个路由目录是否有 loading.tsx + error.tsx？
□ 每个 page.tsx 是否导出了 metadata？
□ layout.tsx 是否是 Server Component（无 'use client'）？
□ layout.tsx 的 metadata 是否已从脚手架默认值改为项目信息？
□ 是否有 src/lib/ 公共层（types, constants, api, storage）？
□ 是否存在跨路由直接 import（如 ../../lottery/store）？
□ 是否有 window.location.href 做内部跳转？
□ 是否有残留的 page.module.css 等脚手架文件？
□ 所有组件 Props 是否有 interface 定义？
□ 是否有 any / console.log？
□ useState 泛型是否显式标注？
```

## 三、项目根目录规范

### 3.1 目标结构

```
anti-huihuan-kaifa/
├── frontend/              # Next.js 前端
├── backend/               # FastAPI 后端
├── docs/                  # 项目文档
```

### 3.2 禁止事项

```
❌ backend/backend/     → 删除多余嵌套
❌ backend/app/（空的） → 删除空壳目录，重构后由 app/ 承载代码
❌ page.module.css      → 清理脚手架残留文件
❌ public/ 下的脚手架 SVG（file.svg, globe.svg 等） → 替换为项目实际资源或删除
```

## 四、变更触发规则

| 触发事件 | 需要检查的项 |
|----------|-------------|
| 新建 `.py` 文件 | → 是否放对了目录（schema? service? endpoint?） |
| 新建 `.tsx` 文件 | → 是否 ≤ 150 行？是否有 Props 接口？是否需要 `'use client'`？ |
| 新建路由目录 | → 是否配套 `loading.tsx` + `error.tsx` + `metadata`？ |
| 新建模块目录 | → 是否创建了 GEB L2 `CLAUDE.md`？ |
| 删除文件 | → 是否更新了 L2 成员清单？ |
| 修改 Schema/类型 | → 前后端对应的类型是否同步？ |
| 跨路由 import | → 是否应该提升到 `lib/`？ |

## 五、快速诊断命令

```bash
# 检查超过 150 行的 .tsx 文件
find frontend/src -name "*.tsx" -exec awk 'END{if(NR>150)print FILENAME": "NR" lines"}' {} \;

# 检查缺少 loading.tsx 的路由目录
for d in frontend/src/app/*/; do [ ! -f "$d/loading.tsx" ] && echo "Missing loading.tsx: $d"; done

# 检查缺少 error.tsx 的路由目录
for d in frontend/src/app/*/; do [ ! -f "$d/error.tsx" ] && echo "Missing error.tsx: $d"; done

# 检查 any 使用
grep -rn ": any\|as any" frontend/src/ --include="*.tsx" --include="*.ts"

# 检查 console.log
grep -rn "console\.log" frontend/src/ --include="*.tsx" --include="*.ts"

# 检查 window.location
grep -rn "window\.location" frontend/src/ --include="*.tsx" --include="*.ts"

# 检查后端 print
grep -rn "^[^#]*print(" backend/ --include="*.py"

# 检查后端嵌套目录
[ -d "backend/backend" ] && echo "❌ backend/backend/ 嵌套存在，需删除"
```
